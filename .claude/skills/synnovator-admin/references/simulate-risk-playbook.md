# Risk Playbook — 业界最佳实践风险场景

本文件定义 8 类风控场景，覆盖企业悬赏和团委竞赛两种场景。
在第 5 章（异常与边界案例库）中，**Enterprise** 档案优先选 5.1–5.5；**Youth-league** 档案优先选 5.3–5.8。
也可根据 `risk_event_toggles` 配置按需开启或关闭。

---

## 场景 R-01：涉密数据误用（Enterprise 重点）

**适用**：enterprise
**触发**：参赛团队在公开帖子或 GitHub public repo 中上传了主办方通过官方渠道分发的测试数据集（含企业内部字段或客户 ID）。
**处理最佳实践**：
1. 平台自动扫描提交物料中的敏感关键词（如内部域名、数据库连接串格式），触发人工复核工单
2. 在1小时内通知主办方安全负责人
3. 冻结该队伍提交，发送书面通知要求立即删除
4. 主办方法务介入，评估 NDA 违约风险
5. 若确认违约，取消参赛资格并保留追责权；若误报则解冻并通知参赛者

**结果模板**：队伍「{队名}」因上传含内部数据字段的样本文件被取消资格；平台同步删除相关内容；主办方法务通知当事人 NDA 违约已记录存档。
**平台策略说明**：NDA 签署时应明确"测试数据仅限本地使用，禁止上传至任何公开仓库"；平台应在提交前提示敏感内容检测结果。

## 场景 R-02：成果抄袭（双场景适用）

**触发**：评审期间，某评审指出参赛作品的核心代码模块与 GitHub 上某开源项目高度重合，但提交说明未注明来源。
**处理最佳实践**：
1. 主办方指定一名独立评审核查代码相似度（工具：Copyleaks / SonarQube SAST + 人工比对）
2. 向被质疑队伍发送书面问询，要求在48小时内提交代码来源说明及贡献证明
3. 如队伍证明是合规引用（MIT/Apache license，已署名），恢复正常评审
4. 如无法自证原创，扣分处理（技术可行性维度清零）或取消资格，视主办方规则决定
5. 结果在公示期同步披露处置结论（不公开具体代码对比内容）

**平台策略说明**：提交时要求填写"主要依赖组件与开源协议"字段；平台在提交截止后自动运行相似度扫描并生成报告供评审参考。

## 场景 R-03：刷票 / 大众评审操控（双场景适用）

**触发**：公示阶段，某队伍获得的大众投票数在12小时内激增，增速异常（>3σ 于同期其他队伍），且来源 IP 高度集中于单一省份。
**处理最佳实践**：
1. 平台风控系统标记异常投票记录，自动降权（不直接删除）
2. 生成异常报告，通知主办方运营负责人
3. 主办方决定是否对该队伍发起问询（书面），要求说明宣传行为
4. 若确认组织刷票，清零该队伍大众评审分；若为自然传播（如爆款文章），则恢复原分
5. 公示期延长24小时并发布声明说明处置原则

**平台策略说明**：大众评审权重不宜超过15%；投票采用实名制+每账号限投1次；设置IP/设备指纹去重。

## 场景 R-04：评审利益冲突（双场景适用）

**触发**：终评结束后，某队伍指出其中一位评审委员与获奖队伍队长存在前同事关系，并在社交媒体上截图提供证明。
**处理最佳实践**：
1. 主办方组成由主办方代表+中立观察员组成的利益冲突核查小组
2. 向该评审委员发送书面问询，要求披露与所有参赛队的关系
3. 审查该评审委员对相关队伍的打分是否存在明显偏差（与其他评审均值对比）
4. 若确认偏差：剔除该评审委员对该队的打分，按剩余评审重新加权计算
5. 公告处置结论，说明利益冲突核查机制，不公开评审具体评分

**平台策略说明**：报名评审时强制填写"与参赛团队/参赛者的关系声明"；评审工作台显示提交说明，方便其主动回避。

## 场景 R-05：知识产权侵权（Enterprise 重点）

**触发**：获奖团队提交的方案中包含主办方竞争对手的专利方法（后经公示期第三方指出），并附上专利号。
**处理最佳实践**：
1. 主办方法务对指控专利进行快速查询（USPTO/CNIPA，2–3个工作日）
2. 若侵权成立：暂停颁奖，要求获奖队伍在5个工作日内提交替代方案或放弃该专利方法
3. 若获奖队伍无法修改：重新评审，从候选队伍中选出替代获奖方
4. 若指控不成立：恢复颁奖流程，向举报方书面说明查询结论

**平台策略说明**：提交说明中增加"技术方案是否涉及已知第三方专利"勾选项；主办方 IP 顾问可在评审期后台对标注专利方案进行快速核查。

## 场景 R-06：内容违规（双场景适用，Youth-league 高发）

**触发**：某参赛者在活动帖子中发表了带有地域歧视性语言的评论，被多名用户举报。
**处理最佳实践**：
1. 平台内容安全模块（关键词过滤+人工复核）在1小时内标记
2. 隐藏违规内容，向当事人发送警告通知，说明违规条款
3. 若为首次：警告+禁言24小时
4. 若为二次及以上：移除参赛资格，不退还报名费（若有）
5. 向举报者反馈处置结果（不公开当事人信息）

**平台策略说明**：活动帖子区应启用 AI 内容审核+人工巡查双通道；Youth-league 场景因参与者年龄层偏低，应设置更严格的内容分级策略。

## 场景 R-07：系统故障导致提交失败（双场景适用）

**触发**：提交截止日（T+70 23:59）前2小时，平台文件上传服务因 CDN 节点故障导致大批用户提交失败，错误率达 34%。
**处理最佳实践**：
1. 监控告警在5分钟内触发，运维团队介入
2. 运营团队在15分钟内发布官方声明，告知延期提交政策（24小时延期窗口）
3. 故障期间，提供备用邮件提交通道（收集 repo URL + 文件）
4. 故障修复后，将备用渠道提交迁移至系统
5. 在公示公告中注明本次故障处置经过，对受影响队伍说明补充证明方式

**平台策略说明**：提交截止前2小时应进入系统稳定期（禁止发布新版本）；提交服务应有降级备案（本地上传 → 对象存储直传）。

## 场景 R-08：参赛资格造假（Youth-league 重点）

**触发**：颁奖后，有匿名举报指出某获奖团队中有成员并非在校学生（团委竞赛要求全员在校大学生），并提供了该成员的 LinkedIn 显示其已工作满2年。
**处理最佳实践**：
1. 主办方向该团队发送书面核查通知，要求提交所有成员的在读证明（7个工作日）
2. 同步冻结奖金发放
3. 若核查属实：取消该团队资格，奖项顺延给下一名；已公示的获奖信息更新公告
4. 若证明在读（如在职研究生符合资格）：解冻奖金，发布澄清公告

**平台策略说明**：报名时应要求上传学籍证明或学校邮箱验证；高奖项团队在颁奖前应进行二次身份核验（学信网接口或人工）。
